Heterogeneous Treatment Effects

Causal calibration for treatment-effect predictions

causalCalibration calibrates treatment-effect scores after you have already trained an HTE model. It implements the workflow from the causal isotonic calibration paper together with pooled cross-calibration, practical alternative calibration maps, and doubly robust diagnostics in Python and R.

Overview

What is this package for?

The package is for the part of the workflow that comes after HTE estimation. You bring raw treatment-effect predictions plus the nuisance estimates needed by your chosen loss. The package then fits a post-hoc calibration map, produces calibrated scores, and can estimate remaining calibration error.

Main workflow

Calibrate an existing treatment-effect model

This package does not compete with your upstream learner. It wraps around it. That makes it easy to add calibration to DR-learners, R-learners, T-learners, forests, boosting, or any other black-box HTE pipeline. 

fit_calibrator(...)
fit_cross_calibrator(...)
diagnose_calibration(...)

Why it matters

A good ranking can still have the wrong effect scale

The paper’s motivation is simple: treatment-effect scores may still be useful for ranking or subgrouping even when they overstate or understate the true effect in the tails. Calibration is the step that corrects that scale.

Calibration

Why calibration matters for CATE models

A raw treatment-effect score can be directionally helpful without being numerically trustworthy. Calibration turns the score into something closer to “units with this prediction really have this average treatment effect.”

Step 1

Bring predictions

Start from a raw HTE score and the nuisance estimates needed to build the calibration target.

Step 2

Fit a map

Learn a monotone or simple recalibration map from raw scores to calibrated scores.

Step 3

Predict and check

Apply the map to new scores and use diagnostics to quantify remaining calibration error.

Scale Correction

Calibration fixes magnitude, not just ordering

If a learner overstates high predicted effects or compresses the middle, calibration can restore a more honest effect scale while preserving the basic ranking structure for monotone methods.

Decision Support

Clearer scores support clearer treatment decisions

When a score is used to target treatment, sign and magnitude both matter. Calibration makes thresholding more transparent because the score is closer to the average effect among units receiving that score.

In the paper’s language, calibration is the property that units with the same prediction have an average treatment effect close to that prediction. That is the conceptual center of this package.

Workflows

Should I use standard calibration or cross-calibration?

This is the first major workflow choice. The answer depends on whether your upstream HTE learner already used cross-fitting and whether you saved the fold-specific prediction objects.

Use standard calibration

Choose fit_calibrator() when you have a single prediction per observation or a separate holdout sample that you want to reserve for calibration.

Inputs: predictions, treatment, outcome, and nuisance estimates.

Use cross-calibration

Choose fit_cross_calibrator() when the learner was trained with cross-fitting and you want to calibrate in sample without carving off a second holdout set.

Inputs: pooled OOF predictions plus an n x K matrix of fold-specific predictions.

How prediction works afterward

The current implementation fits one calibration map on pooled OOF predictions, applies that map to each fold-specific column, and aggregates row-wise by the median.

Prediction contract: predict(cross_calibrator, fold_predictions)

Losses and Methods

What do we implement?

There are two conceptual loss choices and four calibration backends. The loss decides which population you are calibrating for; the method decides the shape of the calibration map.

loss="dr"

Doubly robust calibration for the original study population. This is the natural default when you want calibrated scores to speak to the observed population directly.

loss="r"

R-loss calibration for an overlap-weighted population. This is often more stable under weak overlap, but it answers a different target-population question.

method="isotonic"

The paper-native choice: monotone, nonparametric, and the main method most users should start with.

method="smooth_isotonic"

A smoother monotone map built from the isotonic fit. Useful when the raw staircase is too rough for downstream use.

method="linear"

A nonnegative-slope linear recalibration. This is the simplest low-variance choice and the easiest to interpret.

method="histogram"

Coarse piecewise-constant binning. Transparent and easy to inspect, especially as a diagnostic baseline.

Quickstart

Minimal end-to-end examples

The main idea is the same in both languages: fit a calibrator on raw scores, predict calibrated scores, then diagnose how much calibration error remains.

Python

from causal_calibration import (
    fit_calibrator,
    diagnose_calibration,
)

cal = fit_calibrator(
    predictions=tau_hat,
    treatment=a,
    outcome=y,
    mu0=mu0_hat,
    mu1=mu1_hat,
    propensity=e_hat,
    loss="dr",
    method="isotonic",
)

tau_star = cal.predict(tau_new)

diag = diagnose_calibration(
    predictions=tau_star,
    comparison_predictions=tau_hat,
    treatment=a,
    outcome=y,
    mu0=mu0_hat,
    mu1=mu1_hat,
    propensity=e_hat,
)

Diagnostics are part of the package contract

diagnose_calibration() estimates calibration error with a doubly robust procedure and reports uncertainty using balanced jackknife folds. It can compare raw and calibrated predictions directly through comparison_predictions.
